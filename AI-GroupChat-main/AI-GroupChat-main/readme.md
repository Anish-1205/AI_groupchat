
## Setup and Installation ⚙️

1.  **Clone the repository:**
    ```bash
    git clone https://github.com/nitishkapoor18/AI-GroupChat
    cd AI-GroupChat
    ```

2.  **Install dependencies:**
    ```bash
    npm install
    # or
    yarn install
    ```

3.  **Set up Environment Variables:**
    *   Copy `.env.example` to a new file named `.env`:
        ```bash
        cp .env.example .env
        ```
    *   Edit `.env` and add your API keys:
        ```
        MISTRAL_API_KEY="your_mistral_api_key"
        GEMINI_API_KEY="your_google_ai_api_key"
      
        PORT=3001 # Or any port you prefer
        ```

4.  **Transpile TypeScript (if not using a runtime like `ts-node`):**
    ```bash
    npm run build  
    ```

5.  **Run the Backend Server:**
    ```bash
    npm start   
    npm run dev   
    ```
    The server should start, typically on `http://localhost:3001`.

    *   Ensure the frontend WebSocket client is configured to connect to the backend server's WebSocket URL (e.g., `ws://localhost:3001`).

## How It Works (Backend Orchestration) 📜

1.  **WebSocket Connection:** The Svelte frontend establishes a WebSocket connection with the backend server.
2.  **Debate State:** For each connected client, the backend maintains a `Debate` state object, tracking the topic, history, participants, current turn, rounds, and interaction phase.
3.  **User Initiation:**
    *   The user provides a topic and selects AI personas via the frontend.
    *   A `start_debate` message is sent to the backend.
4.  **Automated Debate Phase (`processNextTurn`):**
    *   The backend orchestrator (`processNextTurn`) manages turn-taking.
    *   It determines the `currentPersona` (either a main participant or the moderator based on `MODERATOR_TURN_INTERVAL` and `participantTurnsThisCycle`).
    *   It constructs a detailed prompt for the `currentPersona` using `buildPromptForPersona`, including persona instructions, topic, recent conversation history, and specific task (e.g., critique, suggest).
    *   It calls the appropriate AI model (`fetchMistral` or `fetchGemini`) via `executeAiTurn`.
    *   The AI's response is added to the history and broadcast back to the client.
    *   This continues until `MAX_DEBATE_ROUNDS_PER_PARTICIPANT` is reached.
5.  **Transition to Interactive Q&A (`transitionToUserInteractionPhase`):**
    *   The backend sets `isPausedForUserInteraction = true` and `automatedPhaseComplete = true`.
    *   A system message informs the user they can now ask questions.
6.  **User Interaction Phase (`handleUserMessageDuringInteraction`):**
    *   The user sends messages (type `user_query` or `user_interjection`).
    *   **Targeted Questions:** If the message starts with `@Persona Name: ...`, the backend routes the question to that specific AI using `buildPromptForUserQuestionToPersona`. After the target AI responds, other participants are prompted to comment on that response via `triggerFollowUpFromOtherParticipants`.
    *   **General Comments:** If no `@` targeting, the Moderator (if present) fields the comment first using `buildPromptForGeneralUserComment`, aiming to facilitate and direct questions. Then, other participants are prompted to respond sequentially via `triggerSequentialResponsesToUserComment`.
    *   **Commands:** User can type `/end` (or use a button sending `user_command: end_session`) to trigger summarization.
7.  **Regeneration & Other Actions:** Users can request to regenerate the last AI response or stop/reset the session.
8.  **Conclusion (`concludeSession` & `summarizeAndConclude`):**
    *   When the session ends, a final summary of the entire debate and Q&A is generated by an AI (typically Mistral) using a comprehensive prompt.
    *   The debate state is reset for the client.

## API Endpoints (WebSocket Messages) 💬

The communication primarily happens over WebSockets. The client sends JSON messages, and the server responds with JSON messages.

**Client to Server:**

*   `{ "type": "start_debate", "payload": { "topic": "...", "persona_ids": ["id1", "id2", ...] } }`
*   `{ "type": "user_query", "payload": { "content": "User's question/comment..." } }` (Used in Q&A phase)
*   `{ "type": "user_interjection", "payload": { "content": "User's interjection..." } }` (Can be used during automated debate)
*   `{ "type": "regenerate_last_ai_turn" }`
*   `{ "type": "seek_convergence" }` (Triggers early transition to Q&A from automated debate)
*   `{ "type": "stop_debate" }` (Hard reset)
*   `{ "type": "user_command", "payload": { "command": "end_session" } }` (Equivalent to typing `/end` for summarization)

**Server to Client:**

*   Each message is an object implementing the `ChatMessage` interface:
    ```typescript
    interface ChatMessage {
      id: string;
      speaker: string; // "User", "System", "NILM Expert (Mistral)", etc.
      content: string;
      timestamp: string; // ISO string
      isThinking?: boolean; // True if this is a placeholder for an AI "typing"
      isRegenerated?: boolean; // True if the content is a regenerated response
    }
    ```

## Key Backend Functions Functions 🔑

*   `processNextTurn`: Orchestrates the automated debate turns.
*   `executeAiTurn`: Handles calling the specified AI model with a prompt.
*   `handleUserMessageDuringInteraction`: Manages user input during the Q&A phase, routing to AIs.
*   `triggerSequentialResponsesToUserComment`: Gets all main participants to respond to a user's general comment.
*   `triggerFollowUpFromOtherParticipants`: Gets other participants to comment after one AI answers a user's targeted question.
*   `buildPromptFor...`: A suite of functions to construct context-aware prompts for different scenarios.
*   `summarizeAndConclude`: Generates the final session summary.

## Future Enhancements / TODO 🚀

*   **More Sophisticated Context Management:** Implement rolling summaries or vector DB lookups for very long conversations to manage LLM context window limits.
*   **Advanced Convergence Detection:** Beyond max rounds, explore AI-driven or semantic similarity-based convergence detection.
*   **Tool Use / Function Calling:** Allow AIs to "call tools" (e.g., web search, code execution, database lookup) for more grounded responses.
*   **User Authentication & Debate History Storage:** Persist debates to a database, allow users to resume or review past sessions.
*   **Streaming AI Responses:** Implement token-by-token streaming from AI APIs to the frontend for a more interactive feel.
*   **Fine-tuning Personas:** Allow more granular control over persona behaviors or even fine-tune smaller models for specific roles.
*   **UI/UX Improvements:**
    *   Markdown rendering for AI responses.
    *   Better visual distinction for different phases of the chat.
    *   Allow user to edit/delete their messages.
*   **Error Handling & Resilience:** More robust retries, fallback mechanisms for AI API calls.
*   **Scalability:** Consider a message queue (e.g., RabbitMQ, Redis Streams) for AI processing if handling many concurrent users.
